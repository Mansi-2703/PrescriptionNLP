1. PIPELINE ARCHITECTURE OVERVIEW
================================================================================

The solution implements a 4-layer progressive NLP extraction pipeline:

LAYER 1 - Rule-Based Extraction (Foundation)
  - Regex-based pattern matching for structured prescription elements
  - 150+ medicine name aliases and abbreviation handling
  - Smart tokenization for fused elements (e.g., "Tab1000mg" → "Tab 1000 mg")
  - Normalized frequency/duration/form field extraction

  Result: Baseline extraction with high precision but variable recall

LAYER 2 - Enhanced Rule-Based Parser (Improvement)
  - Advanced pattern matching for complex prescription formats
  - Clinical instruction extraction (e.g., "at bedtime", "after food")
  - Abbreviation expansion (OD→Once daily, BD→Twice daily, etc.)
  - International prescription format support

  Result: Improved recall and clinical context extraction

LAYER 3 - Smart Hybrid Processing (Confidence-Based)
  - Identifies low-confidence records (>3 null fields)
  - Applies selective improvements only to complex cases
  - Preserves high-quality Layer 2 results (≤3 nulls)
  - Batch LLM processing for structured enhancement

  Result: 442 complex cases identified (4.42% of dataset)

LAYER 4 - Complete Integration (Final Output)
  - Consolidates all layers into unified output
  - Applies batch Gemini LLM processing to low-confidence records
  - Rate-limited API calls (2 req/min, 50 req/day protection)
  - Comprehensive field validation and null handling

  Result: Final JSON output with all improvements applied

6. APPROACH JUSTIFICATION & ENGINEERING DECISIONS
================================================================================

WHY RULE-BASED + LLM HYBRID (NOT PURE LLM)?

1. COST-EFFECTIVENESS
   - Rule-based processing: $0 (local regex)
   - LLM fallback: ~$0.001 per 20 records (batch processing)
   - Pure LLM: ~$0.002 per record = $20+ for full dataset
   
2. TRANSPARENCY & DEBUGGING
   - Rules provide interpretable extraction logic
   - Easy to add custom patterns for domain-specific formats
   - Failures traceable to specific regex patterns

3. ROBUSTNESS
   - Rules don't hallucinate (unlike LLMs)
   - Deterministic output for quality control
   - LLM fallback only for genuinely difficult cases

4. SPEED
   - Batch rule processing: ~1000 records/second
   - LLM batch: ~20 records/second
   - Hybrid approach maximizes efficiency

WHY CONFIDENCE SCORING?

  Not all records benefit equally from LLM processing:
  - High-confidence (≥80%): Rule-based extraction sufficient
  - Low-confidence (<80%): LLM refinement provides value
  
  This selective approach saves API quota while improving quality


SMART BATCH PROCESSING:

  Instead of (wasteful):
    20 API calls × 1 record = 1 record/call
  
  We use:
    1 API call × 20 records = 20 records/call
  
  Result: 20x cost reduction, same improvement
